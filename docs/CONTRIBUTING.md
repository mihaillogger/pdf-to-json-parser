# Правила работы с репозиторием (для людей и ИИ-ассистентов)

> Этот файл — единый источник правды по процессу разработки парсера PDF → JSON.
> **Любой ИИ-ассистент (Claude, Copilot, Cursor, Gemini и т.п.) обязан прочитать
> и соблюдать эти правила перед тем, как что-либо менять в репозитории.**
> Нарушение любого пункта ломает сборку в Docker и окружение у других членов команды.

---

## 0. TL;DR для ИИ (самое важное)

- **НИКОГДА** не пиши `pip install`. Только `uv add <пакет>`.
- **НИКОГДА** не пушь напрямую в `main`. Работай только в ветке `feat/...` или `fix/...`.
- **НИКОГДА** не правь `pyproject.toml` или `uv.lock` руками — их меняет только `uv`.
- Любая новая зависимость → `uv add`, и **изменённый `uv.lock` обязан попасть в PR**.
- Модули **не возвращают сырые `dict`** — только Pydantic-модели из `src/parser/schemas.py`.
- Перед коммитом прогнать `uv run ruff check .` и `uv run mypy src/`. Красные проверки в `main` не вливаются.
- Python ≥ 3.10. Кросс-платформенность (Linux / macOS / Windows).

## 1. Управление зависимостями: только `uv`

Менеджер пакетов — [`uv`](https://docs.astral.sh/uv/). Ручное создание `.venv` и `pip` запрещены.

### Развернуть проект с нуля
```bash
git clone <ссылка_на_репо>
cd pdf-to-json-parser
uv sync          # поднимает venv, ставит нужный Python 3.10+ и все зависимости
```
После этого IDE (VS Code / PyCharm) должна сама подхватить интерпретатор из `.venv`.
Если нет — выбрать его вручную.

> Если `uv` не установлен: `brew install uv` (macOS) или
> `curl -LsSf https://astral.sh/uv/install.sh | sh`.

### Добавить новую библиотеку — ЖЕЛЕЗНОЕ ПРАВИЛО
```bash
uv add <название_библиотеки>     # напр.: uv add scikit-learn
```
Эта команда сама:
1. скачает пакет,
2. обновит `pyproject.toml`,
3. пересчитает `uv.lock`.

**В терминале ЗАПРЕЩЕНО `pip install`.** Любая библиотека, поставленная мимо `uv`,
сломает сборку в Docker и окружение у других.

> В каждый pull request с новой зависимостью **обязательно** включать изменённый `uv.lock`.
> Версии всегда зафиксированы.

|---------------|-----------------------|----------------------------------------|

```bash
git checkout main
git pull

```

## 3. Проверка кода перед Pull Request

ТЗ требует: PEP 8 + type hints + docstrings (Google или NumPy style) для публичного API.

Перед коммитом прогнать обе команды:
```bash
uv run ruff check .     # стиль и синтаксис (линтер)
uv run mypy src/        # статическая проверка типов
```
Если `mypy` ругается на отсутствие типов — исправлять, а не игнорировать.
**Код с красными проверками в `main` не вливается.**

## 4. Pydantic-схемы (`src/parser/schemas.py`) — единый стандарт JSON

`schemas.py` — непререкаемый стандарт того, как выглядит итоговый JSON.

### Главное правило
Функции модулей **не возвращают сырые `dict` или самописные dataclass'ы**.
Импортируй нужную модель из `schemas.py`, передай ей данные — она сама проверит типы.

```python
from parser.schemas import Metadata

meta = Metadata(
    title="Manganese-iron supported on porous iron foam...",
    title_en=None,                 # перевода нет → СТРОГО None
    authors=["Yao, Yunjin", "Liu, Yating"],
    abstract="The development of highly active porous catalysts...",
    keywords=[],                   # нет ключевых слов → пустой список []
    doi="10.1016/j.ces.2025.121219",
    journal="Chemical Engineering Science",
    year=2025,
    metadata_source="crossref",
    metadata_confidence=0.9,
)
```

### Очистка текста зашита в базовый класс
Не нужно вручную `.strip()` или склеивать висячие переносы (`"поверхно-\nстный"`).
Отдаёшь грязный текст — схема сама нормализует:
```python
from parser.schemas import Section

section = Section(
    heading="Introduction",
    level=1,
    content="This is a text with a hanging hy-\nphen and trailing spaces.   ",
    subsections=[],
)
# content станет: "This is a text with a hanging hyphen and trailing spaces."
```

### `ValidationError` — это баг в ВАШЕМ парсере, не в схеме
Частые причины:
- не передали обязательное поле (напр., `abstract`);
- передали `None` туда, где должен быть пустой массив `[]` (напр., `authors`, `figures`);
- перепутали типы (строка вместо числа).

Смотри лог ошибки → находи кривое поле → чини свой алгоритм извлечения.

### Контракт между модулями
Каждый модуль отдаёт **готовые объекты схем** (`Section`, `Table`, `Figure`, `Equation`, ...).
Оркестратор собирает их в финальный `Document` и выдаёт валидный JSON.
Имена и типы полей менять нельзя (см. раздел 4 ТЗ).

## 5. Архитектура (чёткое разделение слоёв)

Каждый блок — отдельный модуль/класс, тестируемый независимо:

| Модуль                 | Зона ответственности                                  |
|------------------------|-------------------------------------------------------|
| `cli.py`               | CLI-интерфейс (typer), маршрутизация одиночный/батч    |
| `schemas.py`           | Pydantic-модели итогового JSON (единый стандарт)       |
| извлечение текста      | сырой текст + координаты + картинки (PyMuPDF)          |
| `metadata.py`          | title / authors / doi / abstract / year / journal      |
| `sections.py`          | рекурсивное дерево секций (`level`, `subsections`)      |
| `figures.py`           | фигуры, панели, bbox                                   |
| `equations.py`         | LaTeX-уравнения, `\tag{N}`, bbox                        |

Внешние API (CrossRef, OpenAlex) и LLM — **только опционально, отключаемые флагом CLI**.
Обязателен оффлайн-режим. В README указать модель/провайдера LLM и стоимость обработки 1 документа.

**Запрещено:** hard-coding шаблонов конкретных журналов из корпуса; ручная правка тестовых PDF.
